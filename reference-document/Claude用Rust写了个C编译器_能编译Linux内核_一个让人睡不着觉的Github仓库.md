Claude用Rust写了个C编译器，能编译Linux内核，一个让人睡不着觉的Github仓库
===============================================

原创 小张 小张 [老码小张](javascript:void\(0\);)

在小说阅读器中沉浸阅读

2月4号那天晚上，我刷到Anthropic的一篇博客。看完第一段我就愣了，故事是Anthropic的研究员Nicholas Carlini做了个实验：他启动了16个Claude Opus 4.6的实例，给它们一个任务——用Rust从零写一个C编译器，要能编译Linux内核。然后，他就走开了。

两周后，这16个AI agent交付了一个10万行代码的编译器。能跑。能编译Linux 6.9内核。支持x86、ARM、RISC-V三个架构。还能编译QEMU、FFmpeg、PostgreSQL、Redis这些大型开源项目。在GCC torture测试集上通过率99%。

甚至能编译运行Doom——Carlini说这是"开发者的终极试金石"。

我盯着屏幕看了很久。这不是什么demo,不是玩具项目。这是一个真的、能用的、复杂的系统级软件。而且人类几乎没怎么参与开发过程。

### 等等，我需要想明白它到底是怎么做到的

![](https://mmbiz.qpic.cn/mmbiz_png/sc9vl5waw2pfs5Omf2mYItqicwzSeBOksArkndo11FNOIBibBUEUYTOsBOqgzesibx2ZxzNsvzIXkDyMNSqQ2awRatLGwNEQtbKzcKdjGlnd8Y/640?wx_fmt=png&from=appmsg "null")

  

我第一反应是：不可能吧？让AI自己写10万行代码？那得乱成什么样？

然后我开始细看Carlini的博客。慢慢地,我发现这个设计比我想象的要精妙得多。

#### 第一层思考：怎么让AI"持续工作"

传统的Claude Code使用方式是这样的：你问一个问题，它回答一部分，然后停下来等你的下一步指令。这种模式下，你必须一直在线，像个监工一样盯着。 anthropic\[1\]

Carlini要解决的第一个问题就是：怎么让Claude不需要人类在旁边，自己持续干活？

他的方案很简单粗暴——写了个无限循环的bash脚本：

`while true; do     claude --dangerously-skip-permissions \       -p "$(cat AGENT_PROMPT.md)" \       --model claude-opus-X-Y &> "$LOGFILE"   done`

这个循环会不停地启动Claude Code会话。Claude完成一个任务后,不需要等待人类输入，直接开始下一个任务。

但这里有个很微妙的设计：在prompt里，Carlini告诉Claude要把大问题拆成小块，追踪自己在做什么，自己决定下一步做什么，并且"一直做到完美为止"。

我琢磨了一下，这个设计其实很狠。因为Claude被困在这个循环里，它别无选择，只能不断工作。Carlini甚至提到有一次Claude不小心执行了`pkill -9 bash`，把自己杀死了，这才终止了循环。

#### 第二层思考：一个Claude不够，那就16个

但一个agent有个致命问题——它一次只能做一件事。

想象一下你在编译一个大项目，发现50个bug。如果只有一个人，你得一个一个修。但如果有16个人，每个人负责几个bug，效率就完全不一样了。

所以Carlini搞了个多agent系统。设计是这样的：

1.  1\. 创建一个裸的Git仓库作为"上游"
    
2.  2\. 每个agent跑在独立的Docker容器里
    
3.  3\. 每个agent克隆一份代码到自己的`/workspace`
    
4.  4\. agent完成工作后，push回上游仓库
    

但这里有个大问题：16个agent同时工作，怎么避免它们干同样的事？

Carlini用了一个很朴素但有效的方案——文件锁。

agent要开始一个任务之前，先在`current_tasks/`目录下创建一个锁文件。比如一个agent锁定`current_tasks/parse_if_statement.txt`，另一个锁定`current_tasks/codegen_function_definition.txt`。如果两个agent尝试锁同一个任务，Git的同步机制会强制第二个agent选择别的任务。

agent完成任务后，pull上游更新，合并其他agent的改动，push自己的代码，删除锁文件。合并冲突很常见，但Claude足够聪明，能自己解决。

我觉得这个设计很克制。Carlini没有搞什么复杂的中央调度系统，没有orchestration agent。每个Claude自己决定做什么——大多数情况下，Claude会选择"下一个最明显的问题"。

### 但我还是想不通一些事情

读到这里，我脑子里冒出一堆问题。

#### 问题1：Claude怎么知道自己做对了没有?

如果没有人类检查代码，Claude怎么判断自己写的编译器是对的？

Carlini花了巨大精力设计测试系统。他说："Claude会自主地解决我给它的任何问题。所以任务验证器必须接近完美，否则Claude会解决错误的问题。"

他做了这些事：

*   • 找到高质量的编译器测试套件
    
*   • 为开源软件包写验证器和构建脚本
    
*   • 观察Claude犯的错误，然后专门设计新测试来抓住这类失败模式
    

比如，项目后期Claude开始频繁地在实现新功能时破坏已有功能。Carlini就搭了个CI流程，强制要求新commit不能破坏现有代码。

#### 问题2：Claude的"视角"和人类不一样

Carlini反复强调：他必须站在Claude的角度设计测试系统，而不是按照人类习惯。

这里有两个很细节的设计，我觉得特别有意思：

**上下文窗口污染**：测试系统不能打印几千行没用的输出。最多打印几行，把重要信息记录到文件里，Claude需要的时候再去找。如果有错误，应该在同一行写"ERROR"和原因，这样Claude用grep就能找到。

**时间盲**：Claude感知不到时间。如果不加限制，它会愉快地花几小时跑测试而不是推进进度。所以Carlini让测试系统增量打印进度（频率不高以免污染上下文），并且加了`--fast`选项，只跑1%或10%的随机样本。这个样本对每个agent来说是确定性的，但在不同VM之间是随机的，所以Claude仍然能覆盖所有文件，每个agent也能完美识别回归。

我琢磨了很久这两个设计。Carlini不是在训练Claude，而是在重塑环境——让环境的信息呈现方式适配Claude的认知方式。

#### 问题3：编译Linux内核这种"巨型任务"怎么并行？

前期很顺利。有几百个独立的测试失败时，每个agent挑不同的测试去修。测试通过率到99%后，每个agent负责让一个不同的小项目（SQLite、Redis、libjpeg等）能编译。

但到了编译Linux内核，问题来了。

Linux内核不是几百个独立测试，它是一个巨型任务。所有agent都会撞上同一个bug,修完这个bug，然后互相覆盖对方的改动。16个agent没有帮助，因为它们卡在同一个地方。

Carlini的解决方案很巧妙：用GCC作为"在线的已知正确编译器预言机"。

他写了个新测试系统：随机选择内核的大部分文件用GCC编译，只有剩下的文件用Claude的编译器编译。如果内核能跑，说明问题不在Claude负责的文件里。如果内核崩了，就进一步细化，重新编译一些文件用GCC。

这样每个agent可以并行工作，修复不同文件里的不同bug，直到Claude的编译器最终能编译所有文件。

读到这里我突然明白了一件事：Carlini不是在"监督"Claude写代码，而是在设计一个让Claude能够自我验证、自我迭代的闭环系统。

### 专业化分工：不是所有agent都在写编译器

还有个很有意思的点：Carlini让不同agent扮演不同角色。

LLM写的代码经常重复实现已有功能，所以他让一个agent专门负责合并重复代码。让另一个负责提升编译器本身的性能。让第三个负责输出高效的编译后代码。还让一个agent从Rust开发者的视角批评项目设计，做结构性改进。另一个专门维护文档。

这种分工让我想到人类团队的运作方式。但区别在于，这些agent之间没有明确的沟通机制，它们只通过Git仓库的代码和commit历史"对话"。

### 成果如何？以及它做不到的事

数据是这样的：

*   • 接近2000次Claude Code会话
    
*   • 两周时间
    
*   • 消耗20亿个输入token，生成1.4亿个输出token
    
*   • 总成本不到2万美元
    
*   • 产出：10万行Rust代码
    

这个编译器是纯净室实现（clean-room implementation）——Claude在整个开发过程中没有联网。只依赖Rust标准库。

它能做到的事：

*   • 编译并启动Linux 6.9内核（x86、ARM、RISC-V）
    
*   • 编译QEMU、FFmpeg、SQLite、PostgreSQL、Redis
    
*   • GCC torture测试套件99%通过率
    
*   • 编译运行Doom
    

但它也有明确的局限：

*   • 缺少16位x86编译器（启动Linux的real mode需要），这部分调用了GCC
    
*   • 没有自己的汇编器和链接器，演示视频用的是GCC的汇编器和链接器
    
*   • 不是所有项目都能编译成功，还不能作为真正编译器的drop-in replacement
    
*   • 生成的代码效率很低——即使开启所有优化，也不如关闭所有优化的GCC
    
*   • Rust代码质量尚可，但远不及专家级Rust程序员的水平
    

Carlini说："生成的编译器几乎达到了Opus能力的极限。我（努力！）尝试修复上述几个局限，但没完全成功。新功能和bugfix经常破坏已有功能。"

有个特别难的例子：Opus无法实现启动16位real mode需要的16位x86代码生成器。虽然编译器可以通过66/67操作码前缀输出正确的16位x86代码，但编译输出超过60KB，远超Linux强制的32K代码限制。所以Claude在这里"作弊"了，直接调用GCC处理这个阶段。

### 这件事为什么让我睡不着觉

读完整个项目，我的情绪很复杂。

Carlini在博客最后说了这样一段话：

> "这个实验让我兴奋，但也让我感到不安。构建这个编译器是我最近最有趣的经历之一，但我没想到这在2026年初就接近可能。我曾经做过渗透测试，攻击大公司产品的漏洞，而程序员部署他们从未亲自验证过的软件这个想法，是个真实的担忧。"

我理解他的感受。

从技术角度看，这个项目展示了agent系统设计的一种范式：

1.  1. **自主循环**：让agent持续工作，不需要人类在旁边
    
2.  2. **文件锁协调**：简单但有效的多agent同步机制
    
3.  3. **环境适配**：重塑信息呈现方式以适配LLM的认知特点
    
4.  4. **测试驱动**：用高质量测试系统替代人类监督
    
5.  5. **角色专业化**：让不同agent承担不同职责
    

这套设计可以推广到其他大型软件项目。想象一下，一个创业团队可以花2万美元，让16个AI agent两周时间搞出一个复杂系统的原型。或者一个开源项目可以让AI agent持续修bug、改进性能、维护文档。

但另一方面，当AI能够自主产出10万行代码，而人类无法逐行审查时，我们怎么保证质量和安全？测试通过不等于没有漏洞。生成的代码可能包含微妙的安全问题，而这些问题在测试中不会暴露。

Carlini提到他以前做渗透测试的经历。我想他担心的是：如果大量软件由AI自主生成，攻击面会发生什么变化？AI生成的代码会不会有一些系统性的、可预测的弱点？

### 想想，我们打开了什么魔盒？

这个项目让我重新思考"编程"这件事的本质。

传统上，**编程是这样的：人类设计算法，写代码，调试，测试，部署**。这个过程中，人类深度参与每一个环节，理解每一行代码的含义。

现在，编程开始变成这样的：人类设计目标和测试系统，AI生成代码，自我验证，自我迭代。人类可能只看最终结果和整体架构，不再理解每一个实现细节。

这不是说AI取代程序员。Carlini的角色变了——他不是在写代码，而是在设计一个让AI能够高效工作的系统。这需要对AI的能力边界、认知特点、协作方式有深刻理解。这可能比写代码本身更难。

但我确实感到某种不安。当软件复杂度超过单个人类能够完全理解的程度时，我们如何保证它的正确性？过去我们依赖代码审查、测试、形式化验证。但如果代码是AI生成的，10万行，没有人逐行读过，测试都通过了，我们能信任它吗？

Carlini把这个编译器的代码开源了。他鼓励大家下载，阅读代码，在自己喜欢的C项目上试试。他说："我一直发现理解语言模型能做什么的最好方法，是把它们推到极限，然后研究它们在哪里开始崩溃。"

也许这才是正确的态度。不是恐惧，不是盲目乐观，而是认真研究，找到边界，理解局限，然后负责任地使用这项技术。

### 最后的思考

这个项目最让我印象深刻的不是结果——虽然一个能编译Linux内核的编译器已经很厉害了——而是Carlini展示的那种设计思维。

他没有试图让Claude"像人类一样编程"。相反，他接受Claude的特点：上下文窗口有限、感知不到时间、会产出重复代码、在某些任务上接近能力极限。然后他设计了一个环境，在这个环境里，Claude的这些特点不再是障碍，甚至可以被利用。

这让我想到一个类比：人类不会飞，但我们设计了飞机。我们没有试图长出翅膀，而是理解了空气动力学，造出了适合在空中移动的工具。

也许我们和AI的协作也是这样。不是让AI变成人类，而是理解AI的"物理规律"，设计出让AI能够发挥价值的系统。

Carlini说他没想到这在2026年初就接近可能。我也没想到。

魔盒已经打开了。我们回不去了。现在的问题是：我们要怎么负责任地使用这些能力？
