# 微信公众号技术文章写作风格指南

> 本文档总结了「老码小张」公众号技术文章的写作风格，供大模型参考学习。

---

## 一、标题设计风格

### 1.1 标题结构模式

**模式一：数字 + 成果 + 价值点**
- "800 行代码，从 while(true) 到状态机：Agent 架构演进实践，扩展性更强"
- "新增 500 行代码，我实现 Agent Skills 渐进式加载，原理一目了然，这掌控感极好"
- "增加 300 行代码，实现了 AI Agent 安全防线，让危险命令需要手工确认"

**模式二：技术点 + 对比 + 结论**
- "200 行 JS vs Python 深度学习模型，两种 AI 去水印的技术路径"

**模式三：惊叹式开头 + 核心事件**
- "离大谱！AI 无法肉身翻墙，所以它造了个 Protocol 让我去干活，时薪 50 刀！"
- "Claude 用 Rust 写了个 C 编译器，能编译 Linux 内核，一个让人睡不着觉的 Github 仓库"

**模式四：设问 + 解决方案**
- "让 AI 拥有'肉身'：这个开源项目用 50 行代码实现了 AI 自拍"
- "让 AI 直接画图？这个 Excalidraw MCP 把我的工作流彻底改了"

### 1.2 标题特点

- **数字量化**：用具体数字（行数、时间、金额）增加可信度和吸引力
- **技术名词**：包含关键技术术语，精准定位受众
- **情绪词汇**：适当使用"离大谱"、"睡不着觉"、"彻底改了"等情绪化表达
- **价值承诺**：标题末尾点明文章价值或成果

## 二、开头风格

### 2.1 场景化开头（最常用）

```
昨天刷 GitHub 的时候，看到两个去 AI 水印的项目。

上周五下午，我又在画系统架构图。

下午本来在摸鱼，群里有个哥们甩了一张图，附带了一句："歪日，人给 AI 干活了。"
```

**特点**：
- 以具体时间、场景切入
- 营造"真实经历"的叙事感
- 快速建立读者代入感

### 2.2 疑问式开头

```
你有没有想过一个问题：
当你跟 AI 聊天时，对方是一个"没有身体"的存在。

我第一反应是：不可能吧？让 AI 自己写 10 万行代码？那得乱成什么样？
```

**特点**：
- 用问题激发读者好奇心
- 引出文章要解决的核心问题

### 2.3 承接式开头（系列文章）

```
回顾上次我们做了什么，上一篇文章中，我把代码从 300 行，增加到了 800 行...

之前几篇文章中我们构建的 agent，其实是一个非常危险的 agent...
```

**特点**：
- 快速回顾上下文
- 自然过渡到新内容

## 三、正文结构风格

### 3.1 标准结构模板

```
【场景引入/问题提出】
→ 【问题分析/痛点拆解】
→ 【解决方案/技术方案】
→ 【实现细节/代码示例】
→ 【效果展示/成果总结】
→ 【延伸思考/未来展望】
```

### 3.2 小标题设计

**层级清晰，用数字编号**：
```
### 一、工具系统重构
#### 1. 工具注册中心
#### 2. editFile：搜索替换
#### 3. lintFile：语法检查

### 二、分层记忆系统
```

**疑问式小标题**：
```
#### 为什么是这三个维度？
#### 为什么是 50/100/5min？
#### 为什么不直接写在系统提示词里？
```

**结论式小标题**：
```
#### 安全白名单设计
#### 多规则命中的处理
#### 确认机制：不是 Y/N 那么简单
```

## 四、语言表达风格

### 4.1 口语化与专业性的平衡

**口语化表达**：
```
"我盯着屏幕看了很久"
"我琢磨了一下"
"说破了其实就这样"
"哈哈，完美复刻"
"港真，如果不考 Cursor..."
```

**专业性表达**：
```
"本质上是一个状态流转问题"
"关注点分离"
"增量优于全量"
"协议比实现重要"
```

**平衡技巧**：在技术分析时保持专业，在叙事过渡时使用口语

### 4.2 第一人称叙事

**高频使用"我"**：
```
"我决定深挖一下"
"我脑子里冒出一堆问题"
"我花了三个小时理解它"
"我第一反应是..."
```

**作用**：
- 增强真实感和可信度
- 建立作者与读者的连接
- 展示思考过程而非仅展示结论

### 4.3 设问与自答

```
"所以，问题是什么呢？"
"答案非常简单，还是上下文，模型的上下文是有限的。"

"为什么是这三个维度？"
"这是经验值。我们观察了正常任务的分布..."

"怎么做好呢，就直接参考：Claude Code 的做法吧"
```

### 4.4 对比与类比

**技术对比**：
```
"图片去水印：问题是简单的"
"视频去水印：问题是复杂的"

"分层记忆 vs Agent Skills"
```

**生活类比**：
```
"就像美团外卖派单一样"
"人类不会飞，但我们设计了飞机"
"我们变成了 Microservices（微服务）"
```

## 五、代码与技术内容呈现

### 5.1 代码展示原则

**关键代码完整展示**：
```javascript
// 核心逻辑，完整展示
async checkAndSummarize() {
  if (userMessageCount <= SUMMARIZE_THRESHOLD) return;
  const messagesToSummarize = messages.slice(this.summarizedUpTo, startIndex);
  // ...
}
```

**非关键代码省略或简化**：
```
// 工具注册中心
const toolRegistry = {};
for (const module of toolModules) {
  // 自动发现注册...
}
```

### 5.2 代码注释风格

```
// 中文注释，解释意图
// 检查是否有多处匹配
// 找不到时给出有用的错误提示
```

### 5.3 技术架构图

- 使用 ASCII 图或截图展示架构
- 配合文字解释每个组件的作用
- 强调设计决策背后的原因

## 六、表格使用风格

### 6.1 对比表格

```
| 场景 | 需求 |
|------|------|
| "卡住了吗？" | 状态可观测 |
| 工具调用死循环 | 限制机制 |
| 前端不同步 | 主动查询状态 |
| 加反思功能 | 灵活的流程控制 |
```

### 6.2 功能列表表格

```
| Tool | 说明 | 安全级别 |
|------|------|----------|
| file_read | 读取文件内容 | readonly |
| file_write | 创建或覆盖文件 | write |
```

## 七、情绪与观点表达

### 7.1 适度情绪化

**正面情绪**：
```
"这掌控感极好"
"真香"
"完美复刻"
```

**负面情绪（批评类文章）**：
```
"玩不起就别玩好吗？"
"这不是服务降级，这是欺诈感"
```

### 7.2 观点表达方式

**直接表达**：
```
"我觉得可能是因为..."
"我认为这个设计很克制"
```

**委婉表达**：
```
"也许..."
"可能..."
"我理解他的感受"
```

## 八、结尾风格

### 8.1 总结式结尾

```
"从 while(true) 到状态机，本质上是从'能跑就行'到'可观测、可控制、可扩展'的转变。"

"800 行代码，换来的是：
1. 用户不再问'卡住了吗'
2. Agent 不再无限循环
3. 加新功能比较容易，只是增加新状态"
```

### 8.2 思考式结尾

```
"魔盒已经打开了。我们回不去了。现在的问题是：我们要怎么负责任地使用这些能力？"

"与其说是 AI 需要我们的身体，不如说是我们终于找到了在 AI 时代的一个 API 接口定义。"
```

### 8.3 互动式结尾

```
"不知道你手痒了没，要不要也从零实现一个自己的 agent 呀？"

"如果是你，你会愿意在这个 AI 的世界里，做一个 Node 吗？"
```

### 8.4 资源链接

```
**GitHub 项目地址**：https://github.com/xxx
**技术博客**：https://xxx

#### 引用链接
[1] xxx: https://xxx
```

## 九、排版规范

### 9.1 段落

- 单段不超过 5 行
- 重要观点独立成段
- 使用空行分隔不同主题

### 9.2 强调

**加粗**：关键概念、重要结论
```
**分层是解决复杂性的关键**
```

**代码格式**：技术术语、命令、文件名
```
`rm -rf`、`package.json`、`Agent`
```

### 9.3 列表

**有序列表**：步骤流程
```
1. 第一步...
2. 第二步...
```

**无序列表**：并列要点
```
- 要点一
- 要点二
```

## 十、文章类型与适配

### 10.1 技术实践类

**结构**：问题 → 方案 → 实现 → 效果
**重点**：代码示例、架构图、设计决策
**示例**：Agent 系列文章

### 10.2 技术分析类

**结构**：现象 → 分析 → 原理 → 思考
**重点**：技术对比、底层原理
**示例**：去水印技术对比、MCP 协议分析

### 10.3 项目介绍类

**结构**：发现 → 是什么 → 怎么做 → 能做什么
**重点**：项目亮点、使用场景
**示例**：Clawra、Excalidraw MCP

### 10.4 观点评论类

**结构**：事件 → 分析 → 观点 → 结论
**重点**：逻辑论证、情感共鸣
**示例**：智谱套餐变更评论

## 十一、写作禁忌

### 11.1 避免的表达

- ❌ 过于学术化的表达："综上所述"、"由此可见"
- ❌ 空洞的套话："具有重要意义"、"值得关注"
- ❌ 过度夸张："颠覆性突破"、"革命性创新"

### 11.2 推荐的替代

- ✅ "所以，答案是..."
- ✅ "这个设计让我想到..."
- ✅ "有意思的是..."

---

## 十二、核心写作原则总结

1. **真实感**：用第一人称叙事，展示真实思考过程
2. **场景化**：从具体场景切入，而非抽象概念
3. **渐进式**：由浅入深，层层递进
4. **对比思维**：通过对比突出重点
5. **代码说话**：关键逻辑用代码展示，而非纯文字描述
6. **适度情绪**：保持专业性的同时，适度表达个人感受
7. **价值导向**：每篇文章都要给读者带来可复用的知识或启发
8. **互动意识**：结尾适当引导读者思考或行动
