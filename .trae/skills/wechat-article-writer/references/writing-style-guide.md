# 写作风格指南精简版

> 本文档为「老码小张」公众号技术文章写作风格的精简参考。

---

## 标题设计模式

### 四种常用模式

| 模式 | 示例 |
|------|------|
| 数字 + 成果 + 价值点 | "800 行代码，从 while(true) 到状态机：Agent 架构演进实践，扩展性更强" |
| 技术点 + 对比 + 结论 | "200 行 JS vs Python 深度学习模型，两种 AI 去水印的技术路径" |
| 惊叹式 + 核心事件 | "离大谱！AI 无法肉身翻墙，所以它造了个 Protocol 让我去干活，时薪 50 刀！" |
| 设问 + 解决方案 | "让 AI 拥有'肉身'：这个开源项目用 50 行代码实现了 AI 自拍" |

### 标题特点
- 数字量化增加可信度
- 技术名词精准定位受众
- 适度情绪词汇增加吸引力
- 结尾点明价值或成果

## 开头写法

### 场景化开头（最常用）

```
昨天刷 GitHub 的时候，看到两个去 AI 水印的项目。

下午本来在摸鱼，群里有个哥们甩了一张图，附带了一句："歪日，人给 AI 干活了。"
```

特点：以具体时间、场景切入，营造"真实经历"的叙事感。

### 疑问式开头

```
你有没有想过一个问题：
当你跟 AI 聊天时，对方是一个"没有身体"的存在。
```

特点：用问题激发读者好奇心，引出核心问题。

## 正文结构模板

### 教程类
```
【场景引入/问题提出】
→ 【问题分析/痛点拆解】
→ 【解决方案/技术方案】
→ 【实现细节/代码示例】
→ 【效果展示/成果总结】
→ 【延伸思考/未来展望】
```

### 行业资讯类
```
【发现/事件】
→ 【是什么】
→ 【怎么做】
→ 【能做什么/意味着什么】
```

### 热点评论类
```
【事件描述】
→ 【分析/多方观点】
→ 【我的观点】
→ 【结论/呼吁】
```

## 语言表达技巧

### 口语化与专业性平衡

**口语化表达**：
- "我盯着屏幕看了很久"
- "我琢磨了一下"
- "说破了其实就这样"
- "哈哈，完美复刻"

**专业性表达**：
- "本质上是一个状态流转问题"
- "关注点分离"
- "协议比实现重要"

**平衡技巧**：技术分析时保持专业，叙事过渡时使用口语。

### 设问与自答

```
"所以，问题是什么呢？"
"答案非常简单，还是上下文，模型的上下文是有限的。"

"为什么是这三个维度？"
"这是经验值。我们观察了正常任务的分布..."
```

### 对比与类比

**技术对比**：
```
"图片去水印：问题是简单的"
"视频去水印：问题是复杂的"
```

**生活类比**：
```
"就像美团外卖派单一样"
"人类不会飞，但我们设计了飞机"
"我们变成了 Microservices（微服务）"
```

## 结尾风格

### 总结式
```
"从 while(true) 到状态机，本质上是从'能跑就行'到'可观测、可控制、可扩展'的转变。"
```

### 思考式
```
"魔盒已经打开了。我们回不去了。现在的问题是：我们要怎么负责任地使用这些能力？"
```

### 互动式
```
"不知道你手痒了没，要不要也从零实现一个自己的 agent 呀？"
```

## 排版规范

- 单段不超过 5 行
- 重要观点独立成段
- 使用空行分隔不同主题
- **加粗**：关键概念、重要结论
- `代码格式`：技术术语、命令、文件名

---

## 写作禁忌

| 避免 | 替代 |
|------|------|
| "综上所述"、"由此可见" | "所以，答案是..." |
| "具有重要意义"、"值得关注" | 具体案例 |
| "颠覆性突破"、"革命性创新" | 适度描述 |
| 直接复制粘贴参考资料 | 重新组织、用自己的话表达 |

---

## 核心原则总结

1. **真实感**：用第一人称叙事，展示真实思考过程
2. **场景化**：从具体场景切入，而非抽象概念
3. **渐进式**：由浅入深，层层递进
4. **对比思维**：通过对比突出重点
5. **代码说话**：关键逻辑用代码展示
6. **适度情绪**：保持专业性的同时，适度表达个人感受
7. **价值导向**：每篇文章都要给读者带来可复用的知识或启发
8. **互动意识**：结尾适当引导读者思考或行动
